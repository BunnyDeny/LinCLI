# cli
致力于实现一个C语言版本的简单，易用，开发者友好型的cli框架

## 开发者视角的命令解析器
命令解析器要完成的任务：

（1）对于命令的每个选项，其都应该有一个参数列表，例如'gcc -o main.o'，这里的-o选项有一个参数main.o，也可以选择不指定任何参数，此时有缺省处理
（2）解析到对应的选项之后，仅仅是赋值某个选项结构体对象的对应字段，该字段本身也是开发者开发期间设计的，但是要遵循合理的规范，例如，规定字段必须要有表示选项名字的成员，以及选项的参数列表（argc，argv），然后每个命令有多个选项，因此需要多个这样的结构体对象，可以考虑传递指针首地址以及对象数量来实现。考虑每个选项都绑定一个默认的预解析函数，用于检查选项参数，以及一些预处理操作，例如提前赋值对应的全局对象的相关信息成员，最终真正的解析函数负责根据这些信息来执行最终的具体操作，这个选项的预解析功能的主要目的是为了检查选项参数以及解析对最终执行环节（由最终的解析函数调用）有用的关键信息，如果某个选项参数不正确则直接返回错误信息，只有所有选项的预解析函数正确执行才会执行最终的解析函数，最终的解析函数主要任务就不是解析具体的选项了，而是根据预解析函数记录的信息来执行最终期望的程序，即本框架将繁杂的字符串解析工作分发给了多个预解析函数，最终的解析函数只负责根据全局结构体记录的信息执行即可，这样大大简化了字符串解析的力

（3）解析函数的实现 当（2）确定好之后，解析函数就可以方便地查看用户输入的各个选项及其对应的参数，为了进一步方便实现解析函数，考虑继续实现一系列方便解析字符串的操作
（4）上述提到的解析函数以及解析选项都是不同命令的个性化配置，所以两者都要交给开发者实现，因此需要提供段注册机制由链接脚本自动收集并编译这些信息，减轻开发者的注册初始化这些步骤的压力，专注于设计业务逻辑的实现细节
（5）用户输入的命令要遵循一定的格式规范才能有效实现（2）目前想到的有：
    - 选项必须以'-'或'--'+英文单词或英文字符的形式给定

(6) (2)提到的命令解析器要通过宏来注册，这个宏决定了命令的字符串格式，以及对应的参数解析器，，参数解析器由用户实现，参数解析器的函数
类型是确定好的，用户无法更改，所以为了提高用户使用参数解析器解析命令的体验，系统会内置一系列解析命令选项内容专用的字符串操作函数集
例如将字符串"1 2 3"解析为一个int数组[1, 2, 3]，将"ab ac bc"解析为字符数组，等等，以及会提供一系列解析失败的处理函数，供用户绑定解析
函数的返回值，例如"a 2 3"试图解析为int数组则解析函数返回-1,解析失败处理函数识别到-1就知道触发了此类型的失败进行相应处理

（7）关于选项：
每个选项包含：
选项名称;
选项的各个值 argv argc字符串数组形式
选项值的类型标识，是一个字符，表示选项的值是（空，字符串，字符串数组 int数组...）等类型，这些类型不能自定义，是系统内置类型
另外，这个类型可以通过字符+数字来制定选项允许有几个参数，例如c2代表此选项是字符数组类型且只允许输入2个字符例如"-t a b"
选项的失败处理（可选），可选的原因是有系统缺省处理，系统缺省处理类似打印错误:"选项...的值解析失败"这样，系统缺省处理主要应对选项类型
的解析失败的情况（实际上对于但个选项来说也只有这一种情况了），例如选项-是c2类型，如果用户输入"-t a b c" 或者"-t ab c"都会报错，前者
是因为选项值的数量不对，后者是因为选项的类型是字符，而第一个元素ac是数组
选项的值要存储位置，是一个void*指针，由于选项类型已经明确了，所以这个泛型指针的类型也就明确了，指针的内存大小必须能容纳选项值的大小
选项要存储的位置的内存大小，用于系统检查选项的值写入此内存是否会导致内存溢出

bool类型：选项是否必需，如果必须，则用户为输此选项会报错
字符串：选项是否依赖其他选项"-t -a"表示此选项必须要求用户输入-t -a选项否则报错

{
    char name[16]; //选项的名字-T,--help等
    enum type; //选项的类型，用于表示选项的参数类型（空，字符串，字符串数组 int数组...）
    int param_num;  //选项类型对应的参数个数，填-1代表无需校验个数
    bad_handlers; //校验失败处理（可选，填入NULL则启用系统缺省处理）
    void* mem；   //如果校验成功了，选项要存储的位置，在解析成功的响应处理中使用这个位置来访问选项的各个参数
    int mem_size; //mem的大小（字节）
    bool is_required; //选项是否必须
}
