# 尾行模式打印支持

在嵌入式交互式 CLI 中，后台代码随时可能通过 `cli_printk` / `pr_*` 宏输出日志。如果此时用户正在命令提示符下输入指令，传统的日志打印会直接覆盖或穿插在当前输入行中，导致命令行内容被破坏、光标位置错乱。

LinCLI 的**尾行模式打印**（Inline Print）机制解决了这个问题：当调度器处于交互输入状态（`scheduler_get_char`）时，`cli_printk` 会自动识别当前上下文，在输出日志前清空当前命令行，打印完日志后再**完整重绘命令提示符、已输入内容以及 Tab 补全候选列表**，光标也会恢复到原来的位置。

---

## 效果对比

### 无尾行模式支持

```
lin@linCli> loglev [INFO] sensor update: 25.3℃
```

后台日志直接覆盖在用户输入的命令上，终端一团糟。

### 有尾行模式支持

```
[INFO] sensor update: 25.3℃
lin@linCli> loglev
```

日志独占一行输出，命令提示符和已输入的内容被完整恢复，用户体验与桌面 Shell 一致。

---

## 更复杂的场景：Tab 补全候选列表

当用户按 `Tab` 键触发候选列表显示后，如果此时后台打印日志，LinCLI 不仅会重绘命令行，还会**重新打印候选列表**，确保显示状态始终一致：

```
lin@linCli> l<Tab>
led    loglevel
lin@linCli> l
```

后台日志插入后：

```
[WARNING] voltage low: 3.1V
lin@linCli> l
led    loglevel
lin@linCli> l
```

候选列表被完整保留，用户可以继续浏览或输入。

---

## 使用方式

**零配置，开箱即用。**

尾行模式打印是 `cli_printk` 的内建行为，开发者不需要额外注册、开启开关或修改配置。只要你的代码通过 `pr_info`、`pr_warn`、`pr_err` 等宏输出日志，框架会自动判断当前是否处于交互输入状态，并决定是否执行清行-打印-重绘的完整流程。

```c
pr_info("temperature: %.1f\r\n", temp);   // 自动适配尾行模式
pr_warn("voltage low!\r\n");            // 同上
```

> **注意**：日志内容建议以 `\r\n` 结尾，这样 `cli_printk` 可以正确处理光标位置。如果省略换行符，日志和命令提示符会连接在同一行，虽然不会破坏输入内容，但可读性较差。

---

## 工作原理简述

1. **状态检测**：`cli_printk` 通过检查调度器当前状态，判断是否在 `scheduler_get_char`（交互输入模式）。
2. **清行**：如果是交互模式，先发送 `\r\033[K`（回车+清除到行尾），清空当前命令行。
3. **输出日志**：执行正常的日志格式化、级别过滤、颜色前缀追加和 FIFO 输出。
4. **重绘**：
   - 如果用户没有触发候选列表，仅重绘命令提示符 + 已输入内容 + 恢复光标位置。
   - 如果用户此前按 `Tab` 显示了候选列表，会根据保存的候选状态重新打印命令候选或选项候选列表，最后再上移光标回到命令行并恢复光标位置。

候选列表的状态（是否存在、类型、参数）由 `cmd_line` 模块内部维护，在以下情况下会被保留：
- 用户输入普通字符
- 用户按退格键

在以下情况下会被清除（不再重绘）：
- 按回车执行命令
- 按上下箭头切换历史记录
- 清屏（Ctrl+L）

---

## 移植注意事项

尾行模式打印依赖 ANSI 转义序列（`\r\033[K`、`\033[D` 等）。在移植到不支持 ANSI 的终端（如某些串口屏、裸数码管）时，该功能会失效，但不会影响日志输出本身——`cli_printk` 仍然可以正常打印日志，只是无法自动恢复命令行内容。

如果你的终端不支持 ANSI，可以通过**弱定义**覆盖 `cli_prompt_print()` 等函数，采用更适合当前硬件的回显策略。
